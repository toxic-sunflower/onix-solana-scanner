1. Цель и границы проекта 
1.1. Цель 
Разработать простой и максимально быстрый сканер, который для каждого поддерживаемого 
Solana-токена независимо получает две котировки: первую цену красного стакана на BingX Futures и 
цену покупки этого же токена через Jupiter. После получения актуальной пары цен система 
рассчитывает процентный спред, сохраняет результат в базу данных, обновляет web-дашборд, строит 
исторический график и отправляет Telegram-сигнал при достижении пользовательского порога. 
1.2. Что обязательно входит в MVP 
●​
Получение Ask 1 из красной части стакана BingX Futures через WebSocket. 
●​
Получение котировки покупки каждого токена через Jupiter отдельным worker. 
●​
Опциональная отдельная прокси для каждого токена. 
●​
Расчет одного показателя Spread % без вычета комиссий, funding и дополнительных моделей. 
●​
Сохранение raw-данных и рассчитанного спреда за последние 72 часа. 
●​
Web-дашборд с карточками токенов, сортировкой и кликабельными кнопками. 
●​
Страница свечного графика спреда с интервалами 5m / 15m / 1h. 
●​
Telegram-уведомления в заданном формате. 
●​
Ссылки на BingX Futures, Jupiter Swap, контракт и внутренний график. 
1.3. Что не входит 
●​
Покупка токена, открытие или закрытие фьючерсной позиции. 
●​
Подключение торговых API-ключей пользователя. 
●​
Расчет комиссий, funding rate, проскальзывания или глубины исполнения. 
●​
Оценка чистой прибыли и гарантированной доходности. 
●​
Автоматическая арбитражная стратегия. 
●​
История более трех суток в основной базе MVP. 
●​
Другие CEX и другие блокчейны. 
Ключевой принцип проекта: ничего лишнего. BingX = только Ask 1. Jupiter = только цена 
покупки. Результат = простой процентный спред.

2.1. Формула 
Spread % = ((BingX Ask 1 - Jupiter Buy Price) / Jupiter Buy Price) × 100 
Пример: BingX Ask 1 = 0.000020 USD, Jupiter Buy Price = 0.000019 USD. Система показывает 
положительный спред. Результат округляется только для отображения; в базе хранится значение с 
полной расчетной точностью. 
2.2. Запрещенные подмены данных 
●​
Не использовать Last Price, Mark Price, Index Price или Bid 1 вместо Ask 1. 
●​
Не использовать среднюю цену нескольких уровней стакана. 
●​
Не использовать цену токена из стороннего агрегатора вместо котировки Jupiter. 
●​
Не рассчитывать спред по новой цене одного источника и заведомо старой цене другого источника. 
●​
Не вычитать комиссии, funding, gas и сервисные сборы.

6.3. Подписки и соединения 
●​
Допускается одно соединение с несколькими подписками либо пул соединений — выбор зависит от

●​

7. Парсер Jupiter 
7.1. Бизнес-требование 
Каждый токен Jupiter обслуживается отдельно. Один медленный или ошибочный токен не должен 
блокировать получение цен по другим токенам. Архитектурное правило: один токен = один 
независимый worker = отдельное состояние соединения/запросов = отдельная прокси при наличии. 
7.2. Получаемая цена 
Для токена запрашивается котировка покупки за заранее заданную базовую сумму. Из ответа 
определяется эффективная цена одной единицы токена. Базовый актив и quote amount задаются 
администратором; для всех пользователей в одном сигнале должны использоваться одинаковые 
параметры, чтобы значения были сопоставимы. 
Jupiter Buy Price = фактически затрачиваемая базовая сумма / фактически получаемое 
количество токенов 
7.3. Транспорт 
Предпочтителен отдельный постоянный streaming/WebSocket-канал на каждый токен, если 
выбранный официальный или согласованный провайдер Jupiter поддерживает такой режим. Если 
текущий официальный Quote API работает по запросной модели, worker реализуется как 
непрерывный низколатентный цикл запросов с HTTP keep-alive. Для бизнеса это остается отдельным 
потоком каждого токена. 
7.4. Требования к worker 
●​
Изолированный timeout. 
●​
Изолированная очередь и счетчик ошибок. 
●​
Поддержка индивидуальной прокси. 
●​
Повторный запрос после временной ошибки. 
●​
Ограничение частоты согласно внешним лимитам. 
●​
Сохранение latency_ms, response_status и provider_timestamp. 
●​
Публикация события JUPITER_QUOTE_UPDATED при успешном обновлении. 
●​
Не публиковать нулевую, отрицательную или математически некорректную цену.

9. Синхронизация котировок и расчет спреда 
9.1. Условие расчета 
Spread % рассчитывается только тогда, когда для токена существуют актуальные значения BingX Ask 1 
и Jupiter Buy Price. Конкретное окно свежести задается в конфигурации. Рекомендуемый стартовый 
параметр — не более 2 секунд между received_at двух источников, затем значение корректируется 
после нагрузочного теста. 
Окно свежести является защитой от ложных свечей и резких выбросов, возникающих при 
сравнении новой цены с устаревшей. 
 
9.2. Алгоритм 
1.​ Получить последнее валидное Ask 1 BingX. 
2.​ Получить последнюю валидную Buy Price Jupiter. 
3.​ Проверить enabled и статус токена. 
4.​ Проверить возраст обеих котировок. 
5.​ Проверить разницу их временных меток. 
6.​ Рассчитать Spread %. 
7.​ Сохранить raw tick. 
8.​ Опубликовать событие для дашборда. 
9.​ Проверить условие Telegram-сигнала.

10. Хранение данных и агрегирование 
10.1. Общий принцип 
Все успешно рассчитанные данные парсера поступают в базу данных. Raw ticks хранятся 72 часа. На 
их основе формируются агрегаты для графика с интервалами 5 минут, 15 минут и 1 час. 
10.2. Рекомендуемая технология 
PostgreSQL с расширением TimescaleDB либо другой time-series storage, обеспечивающий быстрые 
вставки, TTL/retention и агрегации OHLC. Использование конкретной технологии согласуется до начала 
разработки, но логическая модель данных обязательна.
